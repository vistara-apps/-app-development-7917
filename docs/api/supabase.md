# Supabase Integration API

ScholarSift uses Supabase for user authentication, data storage, and real-time functionality. This document describes the Supabase integration and database schema.

## Overview

The Supabase integration provides:

1. User authentication and management
2. Data storage for user profiles and summaries
3. Real-time updates for collaborative features

## Database Schema

### Tables

#### profiles

Stores user profile information.

| Column | Type | Description |
|--------|------|-------------|
| id | uuid | Primary key |
| user_id | uuid | Foreign key to auth.users |
| email | text | User's email address |
| subscription_tier | text | User's subscription tier (free, basic, premium) |
| summaries_used | integer | Number of summaries used in current period |
| summaries_limit | integer | Maximum number of summaries allowed |
| created_at | timestamp | Creation timestamp |
| updated_at | timestamp | Last update timestamp |

#### summaries

Stores paper summaries generated by users.

| Column | Type | Description |
|--------|------|-------------|
| id | uuid | Primary key |
| user_id | uuid | Foreign key to auth.users |
| title | text | Paper title |
| url | text | Paper URL or DOI |
| summary_text | text | Generated summary |
| key_findings | jsonb | Array of key findings |
| created_at | timestamp | Creation timestamp |
| updated_at | timestamp | Last update timestamp |

#### subscriptions

Stores subscription information.

| Column | Type | Description |
|--------|------|-------------|
| id | uuid | Primary key |
| user_id | uuid | Foreign key to auth.users |
| tier | text | Subscription tier |
| status | text | Subscription status |
| start_date | timestamp | Subscription start date |
| end_date | timestamp | Subscription end date |
| payment_provider | text | Payment provider (stripe, paypal) |
| payment_id | text | Payment provider ID |
| created_at | timestamp | Creation timestamp |
| updated_at | timestamp | Last update timestamp |

## Row Level Security (RLS) Policies

Supabase RLS policies ensure that users can only access their own data:

### profiles

```sql
CREATE POLICY "Users can view their own profile"
ON profiles FOR SELECT
USING (auth.uid() = user_id);

CREATE POLICY "Users can update their own profile"
ON profiles FOR UPDATE
USING (auth.uid() = user_id);
```

### summaries

```sql
CREATE POLICY "Users can view their own summaries"
ON summaries FOR SELECT
USING (auth.uid() = user_id);

CREATE POLICY "Users can insert their own summaries"
ON summaries FOR INSERT
WITH CHECK (auth.uid() = user_id);

CREATE POLICY "Users can update their own summaries"
ON summaries FOR UPDATE
USING (auth.uid() = user_id);

CREATE POLICY "Users can delete their own summaries"
ON summaries FOR DELETE
USING (auth.uid() = user_id);
```

## API Endpoints

### User Management

#### Get Current User

```
GET /api/v1/user
```

Returns the current user's profile information.

#### Update User Profile

```
PATCH /api/v1/user
```

Updates the current user's profile information.

### Summaries Management

#### Get User Summaries

```
GET /api/v1/summaries
```

Returns all summaries for the current user.

#### Get Summary by ID

```
GET /api/v1/summaries/:id
```

Returns a specific summary by ID.

#### Create Summary

```
POST /api/v1/summaries
```

Creates a new summary.

#### Update Summary

```
PATCH /api/v1/summaries/:id
```

Updates an existing summary.

#### Delete Summary

```
DELETE /api/v1/summaries/:id
```

Deletes a summary.

### Subscription Management

#### Get Current Subscription

```
GET /api/v1/subscription
```

Returns the current user's subscription information.

#### Update Subscription

```
PATCH /api/v1/subscription
```

Updates the current user's subscription.

## Authentication

Supabase provides multiple authentication methods:

- Email/password
- Magic link
- OAuth (Google, GitHub, etc.)

## Real-time Functionality

Supabase's real-time functionality is used for:

- Updating summary counts in real-time
- Collaborative editing of summaries (premium feature)
- Notifications for subscription status changes

## Error Handling

Supabase-related errors are handled with specific error codes:

- `SUPABASE_AUTH_ERROR`: Authentication errors
- `SUPABASE_DB_ERROR`: Database errors
- `SUPABASE_STORAGE_ERROR`: Storage errors

## Security Considerations

- All database access is controlled through RLS policies
- API keys are stored securely and never exposed to clients
- User data is encrypted at rest
- Authentication tokens are short-lived and rotated regularly
